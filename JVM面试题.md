## 一、CMS 收集器专题

### 1.1 CMS 基本介绍 🔥

**题干**

> 请简单介绍一下 CMS 收集器，它的主要特点和适用场景是什么？

**参考答案**

✅ **CMS（Concurrent Mark Sweep）简介**

> CMS 是 JDK 1.5 引入的老年代垃圾收集器，目标是**最短停顿时间**。

✅ **主要特点**

| 特点 | 说明 |
|------|------|
| 并发收集 | 垃圾回收与用户线程同时运行 |
| 低停顿 | 只有初始标记和重新标记需要 STW |
| 标记-清除算法 | 不会产生内存碎片（但会产生碎片） |
| 老年代收集器 | 通常搭配 ParNew（新生代）使用 |

✅ **适用场景**

- 对**响应时间敏感**的应用（如 Web 服务、电商系统）
- 老年代对象较多，需要低延迟收集
- JDK 8 及以下版本（JDK 9 标记废弃，JDK 14 移除）

---

### 1.2 CMS 回收阶段 🔥

**题干**

> CMS 的垃圾回收过程分为哪几个阶段？哪些阶段会 STW？

**参考答案**

✅ **四个阶段**

| 阶段 | 是否 STW | 说明 |
|------|---------|------|
| **初始标记**（Initial Mark） | ✅ STW | 标记 GC Roots 直接关联的对象，速度很快 |
| **并发标记**（Concurrent Mark） | ❌ 并发 | 从 GC Roots 出发，遍历整个对象图，耗时最长 |
| **重新标记**（Remark） | ✅ STW | 修正并发标记期间变动的标记，比初始标记长但比并发标记短 |
| **并发清除**（Concurrent Sweep） | ❌ 并发 | 清除垃圾对象，释放内存 |

✅ **流程图**

```text
初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除
     │                │            │                │
     ▼                ▼            ▼                ▼
  标记 GC Roots     遍历对象图    修正标记         清除垃圾
```

---

### 1.3 浮动垃圾

**题干**

> 什么是"浮动垃圾"？CMS 是如何处理浮动垃圾的？

**参考答案**

✅ **浮动垃圾定义**

> 在**并发标记**和**并发清除**阶段，用户线程还在运行，会产生新的垃圾。这些垃圾在本次 GC 中无法被标记，只能留到下次 GC 处理，称为"浮动垃圾"。

✅ **CMS 处理方式**

| 措施 | 说明 |
|------|------|
| 预留内存 | 不在老年代满时才触发 GC，预留一部分空间给浮动垃圾 |
| `CMSInitiatingOccupancyFraction` | 设置触发 GC 的阈值（默认 92%） |
| `UseCMSInitiatingOccupancyOnly` | 是否始终使用该阈值 |

> 💡 **红豆提醒**：如果预留空间不够，浮动垃圾无法存放，会触发 **Concurrent Mode Failure**，退化为 Serial Old 收集器喵！

---

### 1.4 CMS 算法选择

**题干**

> CMS 为什么使用"标记-清除"算法而不是"复制算法"或"标记-整理"？

**参考答案**

✅ **原因分析**

| 算法 | 为什么不用 |
|------|-----------|
| **复制算法** | 需要额外的内存空间（如 Survivor 区），老年代对象存活率高，复制成本大 |
| **标记-整理** | 需要移动对象并更新引用，无法与用户线程并发执行 |

✅ **标记-清除的优势**

- 不需要移动对象，可以与用户线程并发执行
- 实现简单，停顿时间短

✅ **缺点**

- 会产生内存碎片
- 碎片过多时会导致分配失败，触发 Full GC

---

### 1.5 Concurrent Mode Failure

**题干**

> 什么是 Concurrent Mode Failure？发生的原因是什么？如何处理？

**参考答案**

✅ **定义**

> CMS 在并发清除阶段，如果老年代空间不足以容纳新晋升的对象，就会触发 Concurrent Mode Failure。

✅ **发生原因**

| 原因 | 说明 |
|------|------|
| 并发清除阶段产生新对象 | 用户线程还在运行，可能产生新对象 |
| 浮动垃圾占用空间 | 上次 GC 遗留的浮动垃圾 |
| 预留空间不足 | `CMSInitiatingOccupancyFraction` 设置过高 |

✅ **处理方式**

> 退化为 **Serial Old** 收集器（单线程、标记-整理），进行一次完整的 Full GC，停顿时间很长。

⚠️ **红豆警告**：Concurrent Mode Failure 是 CMS 最严重的性能问题，一定要避免喵！

---

### 1.6 重新标记阶段 🔥

**题干**

> CMS 的"重新标记"阶段具体做了什么？为什么要做这个阶段？

**参考答案**

✅ **重新标记的作用**

> 修正**并发标记**期间，由于用户线程运行导致的标记变动。

✅ **具体操作**

| 操作 | 说明 |
|------|------|
| 重新扫描 GC Roots | 检查并发标记期间是否有新的 GC Roots |
| 处理写屏障记录 | 扫描并发标记期间被修改的对象引用 |
| 修正标记 | 更新对象的存活状态 |

✅ **为什么需要这个阶段**

> 并发标记期间，用户线程可能修改了对象引用（如将已标记为垃圾的对象重新引用），如果不修正，会导致**误删存活对象**。

---

### 1.7 CMS 与 ParNew 的关系

**题干**

> CMS 和 ParNew 是什么关系？为什么 CMS 要搭配 ParNew 使用？

**参考答案**

✅ **关系**

| 收集器 | 作用区域 | 算法 |
|--------|---------|------|
| **ParNew** | 新生代 | 复制算法 |
| **CMS** | 老年代 | 标记-清除算法 |

✅ **为什么搭配使用**

- CMS 只负责老年代，需要一个新生代收集器配合
- ParNew 是多线程收集器，适合多核 CPU
- ParNew 可以与 CMS 并发执行，减少整体停顿时间

✅ **其他搭配方式**

- **Serial + CMS**：单核 CPU 或小堆场景
- **Parallel Scavenge + CMS**：JDK 8 支持，但不推荐（Parallel Scavenge 与 CMS 的交互较弱）

---

### 1.8 CMS 缺点

**题干**

> CMS 有哪些缺点？你是如何看待这些缺点的？

**参考答案**

| 缺点 | 说明 | 影响 |
|------|------|------|
| **内存碎片** | 标记-清除算法产生碎片 | 碎片过多时触发 Full GC |
| **CPU 敏感** | 并发阶段占用 CPU 资源 | CPU 核心数少时影响吞吐量 |
| **浮动垃圾** | 并发阶段产生新垃圾 | 预留空间不足时触发 Concurrent Mode Failure |
| **无法处理浮动垃圾** | 需要预留空间 | 内存利用率降低 |

✅ **如何看待**

- CMS 的缺点是**以空间换时间**的设计取舍
- 对于响应时间敏感的应用，这些缺点是可以接受的
- JDK 9+ 推荐使用 G1 替代 CMS

### 1.9 CMS 调优参数

**题干**

> 如何调优 CMS？常用的 CMS 参数有哪些？

**参考答案**

✅ **核心参数**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-XX:+UseConcMarkSweepGC` | - | 启用 CMS |
| `-XX:CMSInitiatingOccupancyFraction` | 92 | 触发 GC 的老年代占用阈值 |
| `-XX:+UseCMSInitiatingOccupancyOnly` | false | 是否始终使用该阈值 |
| `-XX:ParallelGCThreads` | CPU 核心数 | 并行阶段的线程数 |
| `-XX:ConcGCThreads` | (ParallelGCThreads+3)/4 | 并发阶段的线程数 |
| `-XX:+CMSScavengeBeforeRemark` | false | 重新标记前先做一次 Young GC |
| `-XX:+CMSParallelRemarkEnabled` | true | 并行重新标记 |

✅ **调优建议**

| 场景 | 调优方向 |
|------|---------|
| 频繁 Full GC | 降低 `CMSInitiatingOccupancyFraction` |
| 重新标记耗时长 | 开启 `CMSScavengeBeforeRemark` |
| 内存碎片严重 | 开启 `UseCMSCompactAtFullCollection`（已废弃） |

### 1.10 CMS vs G1 对比 🔥

**题干**

> CMS 和 G1 相比，各有什么优缺点？你会如何选择？

**参考答案**

**📊 CMS vs G1 对比**

| 特性 | CMS | G1 |
|------|-----|-----|
| 算法 | 标记-清除 | 标记-整理 + 复制 |
| 内存碎片 | 有碎片 | 无碎片 |
| 停顿时间 | 不可预测 | 可预测（`MaxGCPauseMillis`） |
| 堆布局 | 连续内存 | Region 分区 |
| 适用堆大小 | < 8GB | 8GB ~ 数百 GB |
| 并发标记 | 增量更新 | SATB |
| 内存利用率 | 较高 | 较低（RSet 开销） |
| JDK 版本 | JDK 8 及以下 | JDK 9+ 默认 |

✅ **选择建议**

| 场景 | 推荐 |
|------|------|
| JDK 8 + 小堆 + 低延迟 | CMS |
| JDK 9+ + 大堆 + 可预测停顿 | G1 |
| JDK 11+ + 超大堆 + 极低延迟 | ZGC |

## 二、G1 收集器专题

### 2.1 G1 核心设计思想 🔥

**题干**

> 请简单介绍一下 G1 收集器的核心设计思想，它和 CMS 最大的不同是什么？

**参考答案**

✅ **G1 核心设计思想**

> G1（Garbage-First）是 JDK 9 默认的垃圾收集器，目标是**在满足停顿时间的前提下，最大化吞吐量**。

✅ **与 CMS 最大的不同**

| 特性 | CMS | G1 |
|------|-----|-----|
| 内存布局 | 连续的年轻代/老年代 | Region 分区（不连续） |
| 算法 | 标记-清除 | 标记-整理 + 复制 |
| 碎片处理 | 有碎片 | 无碎片（Region 间复制） |
| 停顿预测 | 不可预测 | 可预测（`MaxGCPauseMillis`） |
| 收集范围 | 只收集老年代 | 整个堆（Mixed GC） |

✅ **G1 的 Region 模型**

```text
┌─────────────────────────────────────────────────────────┐
│                      G1 堆内存                           │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐      │
│  │ Eden│ │ Eden│ │Surv.│ │ Old │ │ Old │ │Hum. │      │
│  │     │ │     │ │     │ │     │ │     │ │     │      │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘      │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐      │
│  │ Old │ │ Eden│ │ Old │ │Surv.│ │ Old │ │ Old │      │
│  │     │ │     │ │     │ │     │ │     │ │     │      │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘ └─────┘      │
│                                                          │
│  Eden = 年轻代（Eden）                                    │
│  Surv. = 年轻代（Survivor）                               │
│  Old = 老年代                                             │
│  Hum. = 大对象（Humongous）                               │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Region 设计与可预测停顿 🔥

**题干**

> G1 的 Region 设计有什么好处？它是如何实现"可预测停顿"的？

**参考答案**

✅ **Region 设计的好处**

| 好处 | 说明 |
|------|------|
| 灵活分配 | 每个 Region 可以动态充当 Eden、Survivor、Old |
| 避免碎片 | Region 间复制，不产生内存碎片 |
| 优先收集 | 优先回收垃圾最多的 Region（Garbage-First） |

✅ **可预测停顿实现**

1. **Region 价值评估**：每个 Region 记录回收价值（回收获得的空间 / 回收耗时）
2. **停顿预测模型**：根据历史数据预测本次 GC 的停顿时间
3. **动态选择 Region**：在不超过目标停顿时间的前提下，选择回收价值最高的 Region

✅ **关键参数**

```bash
-XX:MaxGCPauseMillis=200  # 目标停顿时间（默认 200ms）
```

### 2.3 RSet 与跨代引用 🔥

**题干**

> G1 的 RSet 是什么？它是如何解决跨代引用问题的？和 CMS 的卡表有什么不同？

**参考答案**

✅ **RSet（Remembered Set）**

> 每个 Region 都有一个 RSet，记录**其他 Region 指向本 Region 的引用**。

✅ **解决跨代引用问题**

- 不需要扫描整个堆来查找跨代引用
- 只需要扫描 RSet 就能找到所有指向本 Region 的引用
- 大幅减少 GC 时的扫描范围

✅ **RSet vs 卡表**

| 特性 | RSet | 卡表 |
|------|------|------|
| 粒度 | Region 级别 | 卡页（512 字节）级别 |
| 存储内容 | 记录引用来源 | 记录脏卡 |
| 内存开销 | 较大（5%~20%） | 较小 |
| 适用场景 | G1 | CMS |

### 2.4 SATB 算法 🔥

**题干**

> G1 的并发标记采用的是 SATB 算法，而不是 CMS 的增量更新。为什么？SATB 解决了什么问题？

**参考答案**

✅ **SATB（Snapshot-At-The-Beginning）**

> 在并发标记开始时，对整个堆做一个**逻辑快照**。并发标记期间，即使对象引用发生变化，也按照快照中的状态进行标记。

✅ **与增量更新的区别**

| 特性 | SATB | 增量更新 |
|------|------|---------|
| 策略 | 记录删除的引用（写屏障） | 记录新增的引用（写屏障） |
| 问题 | 可能多标（浮动垃圾） | 可能漏标（需要重新扫描） |
| 重新标记 | 较快 | 较慢 |
| 适用场景 | G1 | CMS |

✅ **SATB 解决的问题**

- 减少重新标记阶段的扫描范围
- 避免 CMS 重新标记阶段的长时间停顿
- 代价是可能产生更多浮动垃圾（但 G1 可以在 Mixed GC 中处理）

### 2.5 Mixed GC 与 Young GC

**题干**

> G1 的 Mixed GC 是什么？它和 Young GC 有什么区别？什么时候会触发 Mixed GC？

**参考答案**

✅ **Young GC**

- 只回收年轻代 Region（Eden + Survivor）
- 触发条件：Eden 区满

✅ **Mixed GC**

- 回收年轻代 Region + **部分老年代 Region**
- 触发条件：老年代占用率达到 `InitiatingHeapOccupancyPercent`（默认 45%）

✅ **对比**

| 特性 | Young GC | Mixed GC |
|------|---------|---------|
| 回收范围 | 年轻代 | 年轻代 + 部分老年代 |
| 触发条件 | Eden 区满 | 老年代占用率达标 |
| 停顿时间 | 较短 | 较长 |
| 目的 | 回收年轻代垃圾 | 回收老年代垃圾，避免 Full GC |

---

### 2.6 G1 写屏障

**题干**

> G1 的写屏障有几种？分别有什么作用？

**参考答案**

✅ **两种写屏障**

| 类型 | 作用 | 时机 |
|------|------|------|
| **Pre-Write Barrier** | 在引用修改前，将旧引用记录到 SATB 队列 | 写操作前 |
| **Post-Write Barrier** | 在引用修改后，更新 RSet | 写操作后 |

✅ **具体流程**

```text
对象引用修改：
  1. Pre-Write Barrier：记录旧引用到 SATB 队列（用于并发标记）
  2. 执行写操作：obj.field = newRef
  3. Post-Write Barrier：更新 RSet（用于跨代引用追踪）
```

### 2.7 停顿预测模型

**题干**

> G1 的停顿预测模型是如何工作的？如果实际停顿时间超过了设置的 `MaxGCPauseMillis`，会怎么办？

**参考答案**

✅ **工作原理**

1. G1 记录每个 Region 的回收耗时和回收获得的空间
2. 计算每个 Region 的**回收价值**（空间/耗时）
3. 根据目标停顿时间，选择回收价值最高的 Region 集合（CSet）
4. 如果预测停顿时间超过目标，减少 CSet 中的 Region 数量

✅ **超过目标停顿时间的处理**

| 情况 | 处理方式 |
|------|---------|
| 轻微超时 | G1 会尝试减少下次 GC 的 Region 数量 |
| 严重超时 | 可能触发 Full GC |
| 持续超时 | 需要调优（增大堆、调整参数） |

### 2.8 G1 Full GC

**题干**

> G1 的 Full GC 会在什么情况下发生？和 CMS 的 Concurrent Mode Failure 相比，有什么异同？

**参考答案**

✅ **G1 Full GC 触发条件**

| 条件 | 说明 |
|------|------|
| Mixed GC 无法回收足够空间 | 老年代垃圾太多，Mixed GC 回收速度跟不上 |
| 大对象分配失败 | Humongous 对象无法找到连续 Region |
| 晋升失败 | 年轻代对象无法晋升到老年代 |

✅ **与 CMS Concurrent Mode Failure 对比**

| 特性 | G1 Full GC | CMS Concurrent Mode Failure |
|------|-----------|----------------------------|
| 触发原因 | 回收速度跟不上分配速度 | 老年代空间不足 |
| 退化方式 | 单线程标记-整理 | Serial Old（单线程标记-整理） |
| 停顿时间 | 很长 | 很长 |
| 解决方案 | 增大堆、调优参数 | 降低触发阈值、增大堆 |

### 2.9 G1 调优实战

**题干**

> 如果给你一个 16GB 堆的电商应用，要求 GC 停顿不超过 100ms，你会选择 G1 吗？为什么？你会怎么调优？

**参考答案**

✅ **选择 G1 的原因**

| 原因 | 说明 |
|------|------|
| 堆大小合适 | 16GB 属于 G1 的最佳适用范围（8GB ~ 数百GB） |
| 停顿时间可控 | G1 支持可预测停顿（`MaxGCPauseMillis`） |
| 无碎片问题 | 电商应用对象生命周期复杂，G1 避免碎片 |

✅ **调优参数**

```bash
# 基础配置
-Xms16g -Xmx16g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100

# Region 大小（根据堆大小调整）
-XX:G1HeapRegionSize=16m

# 触发 Mixed GC 的阈值
-XX:InitiatingHeapOccupancyPercent=40

# 并发标记线程数
-XX:ConcGCThreads=4

# GC 日志
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

✅ **调优步骤**

1. 先用默认参数运行，观察 GC 日志
2. 分析停顿时间是否满足要求
3. 如果 Young GC 停顿过长，增大 `MaxGCPauseMillis` 或减小年轻代
4. 如果 Mixed GC 频繁，降低 `InitiatingHeapOccupancyPercent`
5. 如果 Full GC 出现，增大堆或优化代码

### 2.10 G1 优缺点与选型

**题干**

> G1 的优缺点分别是什么？在什么场景下你会选择 G1，什么场景下你会选择 CMS 或 ParallelGC？

**参考答案**

✅ **G1 优点**

| 优点 | 说明 |
|------|------|
| 可预测停顿 | 通过 `MaxGCPauseMillis` 控制停顿时间 |
| 无碎片 | Region 间复制，不产生碎片 |
| 大堆支持 | 适合 8GB 以上的堆 |
| 并发收集 | 降低停顿时间 |

✅ **G1 缺点**

| 缺点 | 说明 |
|------|------|
| 内存开销 | RSet 占用 5%~20% 内存 |
| CPU 开销 | 并发阶段占用 CPU |
| 小堆不适用 | 堆小于 8GB 时优势不明显 |

✅ **选型建议**

| 场景 | 推荐收集器 |
|------|-----------|
| JDK 8 + 小堆 + 低延迟 | CMS |
| JDK 9+ + 中大堆 + 可预测停顿 | G1 |
| JDK 11+ + 超大堆 + 极低延迟 | ZGC |
| 吞吐量优先 + 不关心停顿 | ParallelGC |

### 3.3 GC Roots 判定 🔥

**题干**

> 以下哪些对象可以作为 GC Roots？请解释为什么。
>
> 1. main() 方法中的局部变量
> 2. 静态变量引用的对象
> 3. 正在运行的线程对象
> 4. 被 synchronized 锁住的对象
> 5. JNI 全局引用
> 6. 方法参数中的对象

**参考答案**

✅ **GC Roots 判定**

| 对象 | 是否 GC Roots | 原因 |
|------|--------------|------|
| main() 方法中的局部变量 | ✅ | 虚拟机栈中的局部变量表 |
| 静态变量引用的对象 | ✅ | 方法区中的静态变量 |

### 3.5 G1 核心机制

**题干**

> 解释 G1 中以下术语的含义及作用：
>
> 1. Region
> 2. RSet
> 3. SATB
> 4. CSet
> 5. TAMS
> 6. Mixed GC

**参考答案**

| 术语 | 含义 | 作用 |
|------|------|------|
| **Region** | G1 将堆划分为多个大小相等的 Region | 每个 Region 可以动态充当 Eden、Survivor、Old |
| **RSet** | Remembered Set，记录跨 Region 引用 | 避免扫描整个堆查找跨代引用 |
| **SATB** | Snapshot-At-The-Beginning | 并发标记算法，保证标记正确性 |
| **CSet** | Collection Set，本次 GC 要回收的 Region 集合 | G1 根据回收价值选择最优 Region |
| **TAMS** | Top-At-Mark-Start，标记开始时的指针位置 | 区分并发标记期间新分配的对象 |
| **Mixed GC** | 混合 GC，回收年轻代 + 部分老年代 | 避免老年代空间耗尽触发 Full GC |

### 3.7 类加载机制 🔥

**题干**

> 解释双亲委派模型的工作流程。如果我自己写了一个 `java.lang.String` 类，放在 classpath 下，main 函数里直接使用，会发生什么？为什么？

**参考答案**

✅ **双亲委派模型**

```text
           ┌─────────────────┐
           │ Bootstrap ClassLoader │  ← 加载 rt.jar（java.lang.String）
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │ Extension ClassLoader │  ← 加载 ext 目录
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │ Application ClassLoader │  ← 加载 classpath
           └────────┬────────┘
                    │
           ┌────────▼────────┐
           │ Custom ClassLoader │  ← 自定义加载器
           └─────────────────┘
```

✅ **工作流程**

1. 收到类加载请求，先委派给父加载器
2. 父加载器继续向上委派，直到 Bootstrap ClassLoader
3. 如果父加载器无法加载，才由子加载器尝试加载

✅ **自定义 String 类的情况**

> 会加载 **JDK 自带的 `java.lang.String`**，而不是自定义的。

**原因**：

- `java.lang.String` 属于核心类库，由 Bootstrap ClassLoader 加载
- 双亲委派机制会优先委派给父加载器
- Bootstrap ClassLoader 会加载 `rt.jar` 中的 String 类
- 自定义的 String 类永远不会被加载

> 💡 **红豆提醒**：这是双亲委派的**安全机制**，防止用户篡改核心类库喵！

### 3.10 开放思考题

**题干**

> 有人说"Java 的自动内存管理让程序员不需要关心内存"，你怎么看？结合你学到的 JVM 知识，谈谈自动内存管理的优缺点，以及作为高级工程师应该如何对待内存管理。

**参考答案**

✅ **自动内存管理的优点**

| 优点 | 说明 |
|------|------|
| 降低开发门槛 | 不需要手动分配/释放内存 |
| 避免内存泄漏 | GC 自动回收不再使用的对象 |
| 避免悬空指针 | 引用安全 |
| 提高开发效率 | 专注于业务逻辑 |

✅ **自动内存管理的缺点**

| 缺点 | 说明 |
|------|------|
| GC 停顿 | 影响响应时间 |
| 内存开销 | GC 需要额外的内存（如 RSet、写屏障） |
| 不可控 | 程序员无法精确控制内存回收时机 |
| 性能损耗 | GC 线程占用 CPU 资源 |

✅ **高级工程师应该如何对待**

| 方面 | 做法 |
|------|------|
| **理解原理** | 深入理解 JVM 内存模型、GC 算法、收集器特性 |
| **监控告警** | 建立完善的 GC 监控和告警机制 |
| **调优能力** | 掌握 JVM 参数调优，解决线上问题 |
| **代码优化** | 减少对象分配、避免内存泄漏、合理使用缓存 |
| **技术选型** | 根据业务场景选择合适的收集器和参数 |

> 💡 **红豆总结**：自动内存管理不是"不需要关心内存"，而是"不需要手动管理内存"。高级工程师必须理解内存管理的原理，才能写出高性能、高可用的代码喵！
