## 一、数据结构与底层实现

### 1.1 共同好友功能选型 🔥

**题干**

> 某社交平台需要实现"共同好友"功能：输入两个用户 ID，返回他们共同关注的博主列表，数据量在百万级。
>
> 1. 用哪种数据结构存储每个用户的关注列表？
> 2. 用什么命令查询共同好友？

**参考答案**

✅ 使用 **Set** 存储每个用户的关注列表

- key = `follow:userId`
- value 为 Set of 博主 ID

✅ 查询共同好友使用 `SINTER` 命令

```bash
SINTER follow:1001 follow:1002
```

✅ **为什么不用 Java 循环？**

> `SINTER` 在 Redis 内部以最小集合为基准遍历，复杂度 O(N×M)，且**原子返回**，避免网络多次往返。

**💡 补充知识点**

| 知识点 | 说明 |
|--------|------|
| Set 底层编码 | 小整数集用 `intset`（有序数组，二分查找），大集合用 `dict`（哈希表） |
| SINTER 复杂度 | O(N × M)，N 为最小集合的元素数，M 为集合个数 |
| 持久化结果 | 用 `SINTERSTORE` 存储到新 key |

### 1.2 ZSet 底层编码切换 🔥

**题干**

> 在 Redis 7.0 中，执行以下操作：
>
> ```bash
> ZADD rank 100 "player1"
> ZADD rank 200 "player2"
> ZADD rank 150 "player3"
> ```
>
> 1. 此时 ZSet 的底层编码是什么？
> 2. 如果再添加第 300 个成员，编码会变化吗？变化前后性能有什么影响？

**参考答案**

✅ 当前编码是 `listpack`（Redis 7.0+）或 `ziplist`（旧版）

✅ 编码切换条件

| 条件 | 阈值 |
|------|------|
| 元素数量 | `zset-max-listpack-entries` = **128** |
| 元素大小 | `zset-max-listpack-value` = **64 字节** |

✅ 切换后编码：**skiplist + dict**

**📊 切换前后对比**

| 特性 | listpack（切换前） | skiplist + dict（切换后） |
|------|-------------------|-------------------------|
| 内存 | 连续内存，空间紧凑 | 内存占用增大 |
| 插入/删除 | O(N)（需移动内存） | O(log N) |
| 范围查询 | O(N) | O(log N) |
| 按成员查找 | O(N) | O(1)（dict） |

> ⚠️ **红豆纠错**：阈值是 **128**，不是 300 喵！跳表是通过**多层索引**实现二分查找效果的，不是"缓存"喵！

### 1.3 1亿用户在线状态的内存优化

**题干**

> 需要存储 1 亿个用户的在线状态（0=离线，1=在线），要求内存占用最小，且支持快速查询和批量设置。
>
> 1. 选哪种数据结构？
> 2. 具体怎么实现？

**参考答案**

✅ 使用 **Bitmap（位图）**

```bash
# 标记用户上线
SETBIT online:20260323 123456 1

# 查询用户状态
GETBIT online:20260323 123456
```

✅ 内存计算

> 1 亿用户 = `100,000,000 bit ≈ 12.5 MB`（极其节省！）

✅ 批量操作

- `BITFIELD`：多个位的设置/查询
- `BITOP`：位运算（AND/OR/NOT）

✅ 按天统计：每天一个 key，设置过期时间

> ❌ **红豆纠错**：如果用 String（key = `on_off_status_+userId`），每个 key 至少 40+ 字节开销，1 亿用户需要 **4GB+** 内存，且大量 key 增加 Redis 管理负担。Bitmap 才是正确方案喵！

### 1.4 Hash 内存异常小的原因

**题干**

> 生产环境发现某个 Hash 类型的 key，执行 `HLEN` 显示有 500 个字段，但 `MEMORY USAGE key` 显示只占用了 2KB。
>
> 1. 可能是什么原因？
> 2. 如果是 Redis 6.x 版本，答案会有不同吗？

**参考答案**

✅ 该 Hash 使用了 `listpack`（Redis 7.0+）或 `ziplist`（6.x）编码

- 字段名和值都非常短（如单字母或小整数）
- 内存连续存储，因此占用很小

✅ Redis 6.x 的区别

- `ziplist` 同样紧凑，但存在**连锁更新**隐患

✅ 验证方法

```bash
OBJECT ENCODING key
```

> 💡 **红豆提醒**：这是实际生产中的常见现象，面试官会借此考察是否理解小对象编码喵！

### 1.5 ziplist 的 prevlen 设计及 listpack 改进 🔥

**题干**

> 1. ziplist 的 `prevlen` 字段为什么要设计成可变长度（1字节或5字节），而不是固定 4 字节？
> 2. 这种设计在 Redis 7.0 的 listpack 中是如何改进的？

**参考答案**

✅ prevlen 可变长度设计

| 长度范围 | prevlen 占用 | 说明 |
|---------|-------------|------|
| < 254 字节 | 1 字节 | 绝大多数情况，节省内存 |
| ≥ 254 字节 | 5 字节 | 1 字节标志 + 4 字节长度 |

> 255 被 `zlend` 占用，因此用 254 作为扩展标志。

✅ listpack 改进

- 每个 entry 存储**自己的 len**（当前 entry 总长度）
- **不依赖前一个 entry**，彻底解决**连锁更新**问题
- 从后向前遍历时，通过当前 entry 的 len 字段直接找到前一个 entry

> ✅ **红豆点评**：主人答出了 255 被占用以及 listpack 存储自己的长度，还提到了连锁更新的本质，理解到位喵！ ✧(≖ ◡ ≖✿)

### 1.6 红黑树 vs 跳表（追问）🔥

**题干**

> 为什么 ZSet 用跳表而不用红黑树？

**参考答案**

| 对比维度 | 跳表 | 红黑树 |
|---------|------|--------|
| 实现复杂度 | 代码量小，不易出错 | 旋转操作复杂 |
| 范围查询 | 找到起点后沿底层链表遍历即可 | 需要中序遍历 |
| 内存开销 | 平均 1.33 个指针 | 固定 2 指针 + 颜色位 |
| 并发友好 | 结构简单，更易扩展 | 旋转时需锁整棵树 |

> 💡 **红豆点评**：主人能说出"范围查询更方便"，但未展开内存和并发对比，已属良好喵！

## 二、持久化

### 2.1 RDB 与 AOF 的区别 🔥

**题干**

> Redis 的 RDB 和 AOF 两种持久化方式有什么区别？各自适用什么场景？

**参考答案**

| 特性 | RDB | AOF |
|------|-----|-----|
| 原理 | 定时生成内存快照 | 追加写入操作日志 |
| 数据安全性 | 可能丢失最后一次快照后的数据 | 取决于 fsync 策略 |
| 文件大小 | 紧凑，适合备份 | 较大，但可 rewrite 压缩 |
| 恢复速度 | 快（直接加载） | 慢（重放命令） |
| 适用场景 | 冷备份、灾难恢复 | 对数据安全性要求高的场景 |

**fsync 策略对比**

| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| `always` | 每次写命令都 fsync | 最高 | 最低 |
| `everysec` | 每秒 fsync 一次 | 较高 | 较高（推荐） |
| `no` | 由操作系统决定 | 最低 | 最高 |

> 💡 **生产推荐**：同时开启 RDB + AOF，用 AOF 保证数据安全，用 RDB 做冷备份。

---

### 2.2 AOF 重写机制

**题干**

> AOF 重写是什么？为什么要重写？重写过程中新写入的数据会丢失吗？

**参考答案**

✅ **AOF 重写**

> Redis 创建一个新的 AOF 文件来替代旧的 AOF 文件，新文件只包含恢复当前数据集所需的**最小命令集合**。

✅ **为什么要重写**

- 随着运行时间增长，AOF 文件会越来越大
- 例如对同一个 key 的多次操作，重写后只保留最终状态

✅ **重写过程中新数据不丢失**

1. Redis 会将重写期间的新命令同时写入**旧 AOF 文件**和**重写缓冲区**
2. 重写完成后将缓冲区内容追加到新 AOF 文件
3. 原子性地替换旧文件

## 三、高可用

### 3.1 主从复制原理 🔥

**题干**

> 简述 Redis 主从复制的原理。全量同步和增量同步有什么区别？

**参考答案**

**📥 全量同步（首次连接）**

```text
1. 从节点发送 PSYNC ? -1，请求全量同步
2. 主节点执行 BGSAVE 生成 RDB，同时记录后续写命令到 repl buffer
3. 主节点将 RDB 发送给从节点，再发送 buffer 中的命令
4. 从节点清空旧数据，加载 RDB 并重放命令
```

**📤 增量同步（断线重连）**

```text
1. 从节点发送 PSYNC offset runid，携带上次同步的偏移量
2. 主节点检查 repl backlog 中是否有对应偏移量的数据
3. 若有，发送偏移量之后的数据；若无，降级为全量同步
```

**📊 对比**

| 特性 | 全量同步 | 增量同步 |
|------|---------|---------|
| 触发条件 | 首次连接 / offset 不可用 | 断线重连 |
| 数据量 | 全部数据 | 仅差异部分 |
| 性能开销 | 高（生成 RDB + 网络传输） | 低 |
| 适用场景 | 初始化 | 正常同步 |

### 3.2 哨兵模式

**题干**

> Redis Sentinel（哨兵）的作用是什么？故障转移的过程是怎样的？

**参考答案**

✅ **作用**

| 功能 | 说明 |
|------|------|
| 监控 | 监控主从节点是否正常运行 |
| 故障转移 | 主节点故障时自动将从节点提升为主节点 |
| 通知 | 通知客户端新的主节点地址 |

✅ **故障转移过程**

```text
1. 哨兵检测到主节点主观下线（SDOWN）
2. 哨兵之间协商，达到 quorum 数量后确认客观下线（ODOWN）
3. 哨兵之间选举 Leader Sentinel
4. Leader Sentinel 选择最优从节点（优先级 → offset → runid）
5. 执行 SLAVEOF no one 提升为新主节点
6. 让其他从节点指向新主节点
7. 通知客户端新主节点地址
```

### 3.3 Cluster 集群 🔥

**题干**

> Redis Cluster 的数据分片原理是什么？如何确定某个 key 属于哪个节点？

**参考答案**

✅ **数据分片**

> Redis Cluster 将所有数据划分为 **16384 个槽（slot）**，每个节点负责一部分槽。

✅ **槽位计算**

```text
slot = CRC16(key) % 16384
```

✅ **哈希标签**

使用 `{tag}` 可以让多个 key 映射到同一个槽：

```bash
user:{1001}:name   → slot = CRC16("1001") % 16384
user:{1001}:age    → slot = CRC16("1001") % 16384  # 同一个槽！
```

✅ **为什么是 16384 个槽？**

| 原因 | 说明 |
|------|------|
| 心跳包大小 | 16384 个槽用 2KB 的 bitmap 即可表示 |
| 节点数限制 | 集群节点数通常不超过 1000 个，16384 足够均匀分配 |
| 性能平衡 | 槽太多会增加管理开销，太少会导致数据不均 |

## 四、缓存设计

### 4.1 缓存穿透 🔥

**题干**

> 什么是缓存穿透？如何解决？

**参考答案**

✅ **定义**

> 查询一个**一定不存在**的数据，缓存和数据库都查不到，请求每次都穿透缓存打到数据库。

✅ **解决方案**

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| 布隆过滤器 | 在缓存前加一层布隆过滤器，快速判断 key 是否可能存在 | 内存小，但有误判率 |
| 缓存空值 | 将不存在的 key 也缓存起来（设置较短过期时间） | 简单，但浪费内存 |
| 接口参数校验 | 在接口层拦截明显不合法的请求 | 治标不治本 |

```text
正常请求：  客户端 → 缓存命中 → 返回
穿透请求：  客户端 → 缓存未命中 → 数据库未命中 → 返回空（不缓存则每次穿透）
```

---

### 4.2 缓存击穿 🔥

**题干**

> 什么是缓存击穿？如何解决？

**参考答案**

✅ **定义**

> 某个**热点 key 过期**的瞬间，大量请求同时打到数据库。

✅ **解决方案**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 互斥锁（setnx） | 只允许一个线程去数据库加载数据，其他线程等待 | 数据一致性要求高 |
| 热点数据永不过期 | 不设 TTL，由后台异步更新 | 热点数据 |
| 逻辑过期 | 在 value 中存储过期时间，发现过期后异步更新，先返回旧数据 | 可容忍短暂不一致 |

```text
击穿过程：  热点key过期 → 100个请求同时到达 → 全部打到数据库 → 数据库压力暴增
互斥锁方案：热点key过期 → 1个请求获取锁 → 其他99个等待 → 请求完成后写入缓存 → 其他请求从缓存读取
```

---

### 4.3 缓存雪崩 🔥

**题干**

> 什么是缓存雪崩？如何解决？

**参考答案**

✅ **定义**

> 大量缓存**同时过期**或 **Redis 宕机**，导致请求全部打到数据库。

✅ **解决方案**

| 方案 | 原理 |
|------|------|
| 过期时间加随机值 | 避免同时过期：`TTL = base + random(0, 300)` |
| 多级缓存 | 本地缓存（Caffeine）+ 分布式缓存（Redis） |
| 限流降级 | 数据库前加限流，非核心服务降级 |
| Redis 高可用 | 使用 Cluster 或 Sentinel 保证 Redis 不宕机 |

```text
雪崩过程：  大量key同时过期 → 请求全部打到数据库 → 数据库崩溃 → 整个系统不可用
解决方案：  过期时间加随机值 + 多级缓存 + 限流降级
```

---

### 4.4 缓存与数据库一致性 🔥

**题干**

> 如何保证缓存与数据库的数据一致性？常用方案有哪些？

**参考答案**

✅ **常用方案：Cache Aside Pattern（旁路缓存）**

```text
读：先读缓存 → 缓存未命中 → 读数据库 → 写入缓存
写：先更新数据库 → 再删除缓存（而不是更新缓存）
```

✅ **为什么是删除缓存而不是更新缓存？**

- 避免并发写导致缓存与数据库不一致
- 懒加载思想，下次读取时再重建缓存

✅ **延迟双删（优化方案）**

```text
1. 先删除缓存
2. 更新数据库
3. 休眠一段时间（如 500ms）
4. 再次删除缓存（清除可能被脏数据污染的缓存）
```

## 五、高级特性

### 5.1 Redis 为什么这么快？🔥

**题干**

> Redis 为什么能支持每秒 10 万+ 的 QPS？

**参考答案**

| 原因 | 说明 |
|------|------|
| 纯内存操作 | 数据存储在内存中，读写速度比磁盘快 10 万倍 |
| 单线程模型 | 避免了上下文切换和锁竞争（Redis 6.0 引入多线程 IO，但命令执行仍是单线程） |
| IO 多路复用 | 使用 epoll 实现单线程同时处理多个连接 |
| 高效数据结构 | SDS、ziplist、skiplist 等针对不同场景优化 |
| 通信协议简单 | RESP 协议解析简单高效 |

> 💡 **红豆提醒**：面试时要强调"单线程指的是命令执行是单线程"，Redis 6.0+ 的网络 IO 已经是多线程了喵！

---

### 5.2 内存淘汰策略

**题干**

> Redis 的内存淘汰策略有哪些？生产环境推荐哪种？

**参考答案**

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，内存满时写入报错（默认） |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用的 |
| `allkeys-lfu` | 所有 key 中淘汰最不经常使用的 |
| `allkeys-random` | 所有 key 中随机淘汰 |
| `volatile-lru` | 有过期时间的 key 中淘汰 LRU |
| `volatile-lfu` | 有过期时间的 key 中淘汰 LFU |
| `volatile-random` | 有过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 有过期时间的 key 中淘汰 TTL 最小的 |

> 💡 **生产推荐**：`allkeys-lru`（通用场景）或 `allkeys-lfu`（热点数据明显）。

### 5.3 大 Key 问题 🔥

**题干**

> 什么是大 Key？如何发现和处理大 Key？

**参考答案**

✅ **定义**

| 类型 | 大 Key 标准 |
|------|------------|
| String | 值 > 10KB |
| Hash/List/Set/ZSet | 元素个数 > 5000 个 |

✅ **危害**

- 阻塞其他请求（Redis 单线程）
- 网络带宽消耗大
- 内存不均（Cluster 模式下）

✅ **发现方法**

```bash
# 查找大 Key
redis-cli --bigkeys

# 查找内存占用大的 Key
redis-cli --memkeys
```

✅ **处理方案**

| 方案 | 说明 |
|------|------|
| 拆分 | 将大 Hash 拆成多个小 Hash（如按日期或 ID 取模） |
| 压缩 | 对 String 值进行压缩（如 gzip） |
| 惰性删除 | Redis 4.0+ 支持 `UNLINK`（异步删除） |

## 六、大厂附加题

### 6.1 分布式锁 🔥

**题干**

> 如何用 Redis 实现分布式锁？有什么缺陷？如何优化？

**参考答案**

✅ **基本实现**

```bash
SET lock_key unique_value NX PX 30000
```

| 参数 | 说明 |
|------|------|
| `NX` | 只有 key 不存在时才设置（互斥性） |
| `PX` | 设置过期时间（防死锁） |

✅ **释放锁（Lua 脚本保证原子性）**

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

✅ **缺陷**

| 缺陷 | 说明 |
|------|------|
| 锁过期但业务未完成 | 其他线程可能获取到锁 |
| Redis 主从切换 | 主节点写入锁后宕机，从节点晋升为主节点，锁丢失 |

✅ **优化方案**

| 方案 | 原理 |
|------|------|
| Redisson 看门狗机制 | 后台线程自动续期（默认 30 秒，每 10 秒续一次） |
| RedLock 算法 | 向 N 个独立的 Redis 实例加锁，多数成功才算加锁成功 |

### 6.2 延迟队列

**题干**

> 如何用 Redis 实现延迟队列？

**参考答案**

**方案一：ZSet 实现**

```bash
# 添加延迟消息
ZADD delay_queue <timestamp> <message>

# 消费者轮询
ZRANGEBYSCORE delay_queue 0 <current_timestamp> LIMIT 0 10

# 处理后删除
ZREM delay_queue <message>
```

**方案二：Redisson 的 RDelayedQueue**

- 基于 ZSet + List 封装，提供更简洁的 API
- 支持精确的延迟投递和消费确认
